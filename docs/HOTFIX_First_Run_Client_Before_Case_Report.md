# HOTFIX_First_Run_Client_Before_Case_Report.md
## HOTFIX — تصحيح تسلسل أول استخدام (Client → Case)

## 1. المشكلة المؤكدة

فحص نموذج إضافة القضية (`#modalCase`) يؤكد أن حقل الموكل (`#fCaseClient`) هو
`<input type="hidden">` يُملأ حصراً عبر `clientSelectorList`، وهي قائمة مبنية
من `data.clients` (الموكلين المسجَّلين فعلياً) — لا يوجد إدخال حر لاسم موكل
غير مسجَّل. أي: **لا يمكن إنشاء قضية فعلياً قبل وجود موكل واحد على الأقل** —
كانت رسالة "ابدأ بإضافة أول قضية" تقود المستخدم لخطوة مستحيلة التنفيذ عملياً
(قائمة الموكلين ستكون فارغة عند فتح النافذة).

## 2. الملفات المعدَّلة

| الملف | التعديل |
|---|---|
| `index.html` | إعادة صياغة `#dashboardWelcome` إلى حالتين فرعيتين (`#welcomeStepClient` / `#welcomeStepCase`)؛ ونفس المنطق داخل `#casesEmpty` (`#casesEmptyStepClient` / `#casesEmptyStepCase`) |
| `js/modules/dashboard.js` | `renderDashboard()`: إضافة تبديل بسيط (`style.display`) بين الحالتين حسب `data.clients.length`، داخل نفس فرع "لا توجد قضايا" الموجود مسبقاً — لم يتغيّر أي حساب آخر في الدالة |
| `js/modules/cases.js` | `renderCases()`: نفس منطق التبديل داخل فرع `!rows.length` الموجود مسبقاً — لم يتغيّر استدعاء `casesRepository.search()` ولا أي سطر آخر |

لم يُفتح أو يُعدَّل: أي ملف Repository / DatabaseService / StorageAdapter /
UndoManager، ولا `js/modules/settings.js` (Google Sync)، ولا أي ملف Cache —
مؤكَّد عبر مقارنة checksum كاملة مع النسخة السابقة المُسلَّمة
(`Master_v12_5_UX02.zip`): الفروقات تقتصر على `index.html`,
`js/modules/dashboard.js`, `js/modules/cases.js` فقط.

## 3. منطق الانتقال بين الحالتين

منطق واحد بسيط، مطبَّق في مكانين (لوحة التحكم وشاشة القضايا)، بدون أي دالة
جديدة — فقط قراءة `data.clients.length` (البيانات موجودة أصلاً في الذاكرة،
لا استعلام جديد):

- **`data.clients.length === 0`** (لا يوجد أي موكل بعد) → تُعرض:
  **"ابدأ بإضافة أول موكل"** + زر **"➕ إضافة موكل جديد"** الذي يستدعي
  `navigate('clients');openAddModal();` — نفس الدالة العامة الموجودة أصلاً
  (`openAddModal()` تفتح نافذة "إضافة موكل" تلقائياً حين تكون الصفحة الحالية
  `clients`، دون أي تعديل عليها).
- **`data.clients.length >= 1` و `data.cases.length === 0`** → تُعرض:
  **"ابدأ بإنشاء أول قضية"** + زر **"➕ إضافة قضية جديدة"** الذي يستدعي نفس
  `openAddModal()` (على صفحة `cases`) كما كان الحال سابقاً.
- بمجرد وجود قضية واحدة على الأقل (`data.cases.length >= 1`)، تختفي لوحة
  الترحيب بالكامل كما كان معمولاً به من قبل — لم يتغيّر هذا الشرط.

هذا التبديل يتكرر بنفس الشكل تماماً في مكانين — لوحة التحكم الرئيسية
(`#dashboardWelcome`)، وشاشة القضايا عند خلوّها (`#casesEmpty`) — بحيث لا
يستطيع المستخدم الوصول لأي واجهة تطلب منه إنشاء قضية قبل وجود موكل، بغضّ
النظر من أين دخل.

## 4. التحقق

- **Syntax:** `node --check` على `dashboard.js` و`cases.js` — ناجح. فحص
  توازن وسوم `index.html` بالكامل (Python `html.parser`) — 0 أخطاء.
- **Behavior (محاكاة Node معزولة، بدون متصفح):**

  | العملاء | القضايا | لوحة التحكم: `stepClient` | `stepCase` |
  |---|---|---|---|
  | 0 | 0 | ظاهر | مخفي |
  | 2 | 0 | مخفي | ظاهر |
  | 2 | 1 | (لوحة الترحيب مخفية بالكامل) | — |
  | 0 | 1 | (لوحة الترحيب مخفية بالكامل) | — |

  نفس الأربع سيناريوهات اختُبرت لمنطق `casesEmpty` بمعزل، بنفس النتائج.

- **بحث شامل في المشروع** عن أي بقايا للعبارات القديمة
  ("أول قضية"، "إنشاء أول قضية"، "ابدأ بإضافة قضية"، "Start/Add First Case"):
  لم يتبقَّ أي ظهور قديم — الظهورات الوحيدة المتبقية هي نص الحالة الجديدة
  الصحيحة ("ابدأ بإنشاء أول قضية") ضمن الحالة الثانية (بعد وجود موكل)، وهو
  النص المطلوب صراحة في هذا الـ Hotfix.
- **نطاق التعديل:** مؤكَّد بمقارنة checksum كاملة — 3 ملفات فقط تغيّرت
  (`index.html`, `dashboard.js`, `cases.js`)، ولا تأثير على أي ملف
  Repository/Database/Undo/Cache/Sync.

## 5. النتيجة

**PASS** — أول استخدام يقود المستخدم الآن دائماً إلى "إضافة موكل" أولاً حين
تكون القائمتان فارغتين، ثم إلى "إضافة قضية" بعد وجود موكل واحد على الأقل —
في كل من لوحة التحكم وشاشة القضايا، دون أي تغيير في أي منطق تخزين أو مزامنة.
