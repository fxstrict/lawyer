## 2. AUDIT SUMMARY (قبل التعديل — قراءة حية للكود الفعلي)

- **كيف يعمل HistoryPanel الآن:** `js/core/HistoryPanel.js` يجمّع 9 كائنات
  `UndoManager` مستقلة (واحد لكل كيان) عبر `repo.getUndoManager().exportHistory()`،
  يجمّعها فى Bulk groups، ويرتّبها زمنيًا فى موجزين منفصلين (`undo`/`redo`)، الأحدث
  أولًا. لا يوجد فيه أى منطق بحث أو تصفية — وهذا متروك للطبقة الأعلى (UI) عن قصد.
- **جلب العمليات:** `getFeed()` تُستدعى من `historypanel-ui.js` عند كل `renderHistoryPanel()`.
- **رسم القائمة (قبل):** `historypanel-ui.js` (نسخة 12.6) كانت تستخدم
  `undoBody.innerHTML = feed.undo.map(...).join('')` و`redoBody.innerHTML = ...` —
  إعادة كتابة DOM بالكامل فى كل استدعاء، لِـ **قائمتين منفصلتين** (`#hpUndoList`،
  `#hpRedoList`) يتم التبديل بينهما عبر CSS class `.active` (وليس تصفية حقيقية).
- **مكان إنشاء العناصر:** داخل `renderGroupRow()` — سلسلة نصية HTML واحدة لكل صف،
  مبنية من الصفر فى كل مرة.
- **مكان تحديثها:** لا يوجد تحديث جزئى — كل شىء يُعاد بناؤه.
- **Search:** غير موجود.
- **Tabs:** موجودة لكن بمعنى مختلف تمامًا (Undo/Redo فقط، وليست حسب الكيان).
- **Badge:** موجودة داخل اللوحة نفسها بعد فتحها (`hpUndoCount`/`hpRedoCount`)، لكن
  غائبة تمامًا عن الزر الدائم الظهور فى التوب بار.
- **Incremental Rendering:** غير موجود.

## 4. تفصيل كل بند من التكليف

### Search (البند 5)
حقل `#hpSearchInput` أعلى اللوحة، بلا زر، يعمل على `input` event (لحظى مع كل
ضغطة مفتاح). البحث غير حساس لحالة الأحرف (`toLowerCase()` على الطرفين). يبحث فى
نص فعليًا مُركَّب لكل صف (`searchBlob`) يضم: العنوان المعروض (يشمل اسم الكيان،
نوع العملية، واسم/رقم السجل من `anchor.label` الذى يستخرجه `HistoryPanel.js`
نفسه من `labelFields` المعرّفة لكل كيان — رقم القضية، اسم الموكل، اسم المستند،
اسم المهمة، ... إلخ)، الوقت النسبى والمطلق، تسمية نوع العملية بالعربى، ونص JSON
الخام للسجل نفسه (يغطى أى حقل ظاهر أو غير ظاهر بشكل مباشر). لا يُعاد جلب أو حساب
البيانات — الفلترة تتم على مخرجات `getFeed()` الموجودة أصلًا.

### Tabs (البند 6)
12 تبويبًا: الكل / القضايا / الموكلون / الجلسات / الأطفال / الأتعاب / المهام /
المستندات / المكتبة / الصيغ / Undo / Redo — بالترتيب المطلوب حرفيًا فى التكليف.
تبويبات الكيانات تُصفّى الموجز المدمج (Undo+Redo معًا) حسب `group.entity.key`؛
تبويبا Undo/Redo يصفّيان حسب الاتجاه فقط (كل الكيانات). "الكل" يعرض كل شىء. لا
إعادة بناء بيانات — نفس تعليمة `getFeed()` الأصلية تُستدعى مرة واحدة، والتصفية
JS بحتة (`Array.filter`) على المصفوفة الناتجة.

### Badge التوب بار (البند 7)
`#hpTopbarBadge` (span داخل `#historyPanelBtn` نفسه، الظاهر دائمًا فى التوب بار
وليس فقط بعد فتح اللوحة) يعرض `feed.counts.undo` (إجمالى عدد العمليات القابلة
للتراجع حاليًا عبر كل الكيانات التسعة). يتحدّث **بغضّ النظر عن كون اللوحة مفتوحة
أو مغلقة** — دالة `afterEntityAction()` الجديدة تستدعى `updateTopbarBadge()` دائمًا
بعد أى من الدوال الـ45 الملفوفة (حفظ/حذف/استرجاع/تراجع/إعادة)، وتستدعى إعادة رسم
الجسم الكامل فقط إذا كانت اللوحة مفتوحة فعلًا — توفيرًا للعمل غير الضرورى.

### Incremental Rendering (البند 8)
الممنوع صراحةً (`list.innerHTML=` الكامل فى كل تحديث) أُزيل بالكامل. الآلية
الجديدة (`diffRender()` + `buildRowElement()`/`patchRowElement()`):
- كل صف له مفتاح ثابت (`data-key`) مُشتق من (الاتجاه، الكيان، نوع العملية،
  الطابع الزمنى للمرساة، bulk أم لا وعدد أعضائها). طالما لم يتغيّر شىء منطقيًا
  فى ذلك الصف، مفتاحه لا يتغيّر.
- عند كل `renderHistoryPanel()`: العناصر التى اختفى مفتاحها من القائمة الجديدة
  تُحذف (`el.remove()`) فقط هى. العناصر التى مفتاحها لا يزال موجودًا **لا تُعاد
  كتابتها** — فقط نصّ الوقت النسبى/العنوان يُحدَّث إن تغيّر فعليًا (مقارنة نصية
  قبل الكتابة، تجنّبًا لإعادة تخطيط لا داعى لها). العناصر الجديدة فقط تُبنى من
  الصفر. الترتيب يُصحَّح بـ `insertBefore` فقط عند الحاجة الفعلية (لا نقل بلا سبب).
- **إثبات عملى:** `js/tests/verify_historypanel_ui_completion.js` §4 يمسك مرجع
  DOM node الفعلى لكل صف، يعيد الرسم ببيانات غير متغيّرة، ثم يتحقق أن **نفس
  المرجع بالضبط** ما زال هو نفس العنصر فى الصفحة (وليس نسخة جديدة تشبهه) — هذا
  إثبات مباشر لعدم وجود إعادة بناء كاملة، وليس ادّعاءً غير مُتحقَّق منه.
- **الأداء (قياس تقريبى إرشادى عبر jsdom، وليس متصفحًا حقيقيًا — الأرقام المطلقة
  تختلف فى متصفح حقيقى لكن الاتجاه النسبى صحيح):** محاكاة 450 صفًا (الحد الأقصى
  الموثَّق: 50 × 9 كيانات) عبر 20 إعادة رسم متتالية (يحاكى فتح اللوحة أثناء جلسة
  عمل نشطة). النهج القديم (إعادة بناء كاملة): **~129 مللي ثانية/رسمة**. النهج
  الجديد (Patch تدريجى): **~44 مللي ثانية/رسمة**. تحسّن ~3× فى هذا القياس
  التقريبى، مع ملاحظة أن الفارق الفعلى فى متصفح حقيقى (بدلاً من jsdom) يكون عادة
  أكبر لصالح النهج التدريجى بسبب تكلفة إعادة التخطيط (reflow) والرسم (paint)
  الحقيقية التى لا يحاكيها jsdom بدقة.

### الاتجاه من اليمين (RTL) (البند 9)
`#historyPanel{right:0; transform:translateX(100%)}` (كانت `left:0` /
`translateX(-100%)`) — الآن تنزلق من اليمين لتوافق `dir="rtl"` واتفاقية الـ
Sidebar القائمة فعليًا فى نفس المشروع (`.main{margin-right:268px}` فى
`css/layout.css`، والـ Sidebar نفسه على اليمين). الانتقال يبقى `transform` +
`opacity` عبر `var(--transition)` الموحّد فى المشروع (`ease` من نوع
`cubic-bezier`، بلا اهتزاز — تم اختباره بصريًا عبر مراجعة الكود، لا يوجد أى
`@keyframes` أو تذبذب فى القيم).

### Empty State (البند 10)
نص محسَّن يتغيّر حسب السياق (بدل نص ثابت واحد):
- لا بحث، لا تصفية: "لا توجد عمليات حتى الآن" + "سيظهر هنا سجل جميع التعديلات
  التى تقوم بها داخل البرنامج."
- بحث بلا نتائج: "لا توجد نتائج" + رسالة تذكر نص البحث المُدخَل.
- تبويب Redo فارغ: رسالة خاصة به.
- تبويب كيان فارغ: رسالة خاصة به.
مع أيقونة مناسبة لكل حالة (🕘 / 🔎).

### Timeline محسَّن (البند 11)
كل صف يعرض: Badge نوع العملية بالعربى (إنشاء/تعديل/حذف/استرجاع — تمت ترجمته من
الإنجليزى الذى كان معروضًا سابقًا، دون تغيير `HistoryPanel.js` نفسه، الترجمة فى
طبقة العرض فقط)، اسم الكيان + أيقونته فى Badge منفصل صغير (`hp-entity-badge`)،
اسم/رقم السجل (من `anchor.label`)، الوقت النسبى (نفس منطق 12.6، غير مُعاد
كتابته)، ولون مميز لكل نوع عملية (كما كان فى 12.6: أخضر/أزرق/أحمر/برتقالى —
محفوظ بالكامل).

### الأداء / O(1) (البند 12)
كما فى §Incremental أعلاه: كل صف غير متغيّر يُكلّف مقارنة نصية واحدة أو اثنتين
(`===` على `textContent`) بدل إعادة بناء HTML — عمليًا قريب من O(1) لكل صف غير
متغيّر، والتكلفة الحقيقية تتركّز فقط على الصفوف التى تغيّرت فعلًا (عادة صف واحد
جديد بعد كل عملية مستخدم).

### Responsive (البند 13)
`@media(max-width:640px)` (فى `components.css`، من 12.6 الأصلية، محفوظة) تُوسّع
اللوحة لعرض الشاشة كاملة. أُضيف اكتشاف/إصلاح مشكلة حقيقية: زر فتح اللوحة نفسه
كان مخفيًا بالكامل تحت 768px بسبب قاعدة عامة موجودة مسبقًا فى `responsive.css`
(`display:none` على كل أزرار `btn-ghost`) — أُضيف استثناء دقيق له وحده (راجع
الجدول أعلاه). صف التبويبات الاثنى عشر يلتف (`flex-wrap:wrap`) بدل التمدد
الأفقى القسرى، مع `max-height` + تمرير رأسى داخلى عند الحاجة، لتفادى كسر التخطيط
على الشاشات الضيقة.

### Accessibility (البند 14)
- `Escape` يغلق اللوحة (مستمع عام على `document`، يتحقق أولًا أن اللوحة مفتوحة).
- التبويبات: `role="tablist"`/`role="tab"`/`aria-selected`، وتنقّل بالأسهم
  (`ArrowLeft`/`ArrowRight`) بين التبويبات مع `roving tabindex`.
- اللوحة نفسها: `role="dialog"` `aria-modal="true"` `aria-labelledby`
  `aria-hidden` (يتبدّل مع الفتح/الإغلاق).
- إدارة التركيز: عند الفتح، التركيز ينتقل تلقائيًا لحقل البحث؛ عند الإغلاق،
  يعود التركيز للعنصر الذى كان عليه التركيز قبل الفتح (عادة زر التوب بار نفسه).
- `:focus-visible` واضح على التبويبات، زر الإغلاق، أزرار "العودة/الإعادة إلى
  هنا"، وحقل البحث.
- القائمة (`#hpList`) تحمل `aria-live="polite"` لإعلام قارئات الشاشة بالتحديثات.

## 5. ملاحظات ومخاطر متبقية

- **jsdom كأداة اختبار فقط:** ملف الاختبار الجديد يعتمد على حزمة `jsdom` (مثبَّتة
  محليًا فقط أثناء التحقق عبر `npm install jsdom --no-save`، ثم أُزيلت
  `node_modules` قبل التسليم). هذا **ليس** تبعية إنتاجية جديدة للتطبيق نفسه —
  التطبيق يبقى Vanilla JS بلا أى build step أو حزم خارجية فى runtime، تمامًا كما
  كان. من أراد إعادة تشغيل هذا الـ harness مستقبلًا يحتاج `npm install jsdom`
  محليًا أولًا (لا يوجد `package.json` فى المشروع؛ التثبيت غير محفوظ عمدًا).
- **الترتيب داخل تبويب "الكل":** يُبقى نفس ترتيب `getFeed()` (undo كتلة ثم redo
  كتلة، كل منها الأحدث أولًا) بدل دمجهما زمنيًا فى تسلسل واحد متداخل — قرار تصميمى
  متعمّد لتفادى أى حساب/دمج إضافى غير موجود أصلًا فى `getFeed()` (التزامًا بشرط
  "لا تعيد بناء البيانات" فى التكليف). الأثر: ضمن تبويب "الكل"، كل عناصر Undo
  تظهر قبل كل عناصر Redo، بدل تشابك زمنى كامل. تجميلى/عرضى فقط، ولا يمسّ صحة أى
  عملية جمع أو تراجع.
- **الملاحظتان الثانويتان من تقرير 12.6 الأصلى** (Race نظرى عند نقرات متزامنة على
  أزرار مختلفة؛ ترتيب عرض Redo حسب timestamp الإنشاء لا ترتيب التراجع) لم
  تُعالَجا فى هذه المرحلة عمدًا — هما خارج نطاق "استكمال الواجهة" المحدَّد صراحة
  فى تكليف 12.6A (لا منطق Undo/Redo تم لمسه)، وتأثيرهما موصوف أصلًا بأنه منخفض
  جدًا فى الاستخدام الطبيعى (تطبيق مستخدم واحد Offline-first).
